# Admin Web 整站页面优化设计

## 1. 背景

当前 admin-web 已具备完整业务模块与菜单结构，但不同页面在布局骨架、筛选区、操作区、表格密度、卡片层级、详情信息表达上存在风格分散的问题。用户希望在不改变现有菜单信息架构与核心业务流程的前提下，对整站页面做统一优化设计，并按照现有菜单模块逐步落地。

本次设计以“标准后台简洁”为统一方向，强调列表优先、信息清晰、关键操作突出、长期使用舒适。

## 2. 目标

- 统一全站后台页面视觉骨架
- 统一同类型页面的布局模式与交互节奏
- 保留现有菜单结构、路由关系与核心业务逻辑
- 让后续模块改版可以基于同一套样板复制推进
- 优先适配高频管理场景，兼顾 PC 与移动端

## 3. 非目标

- 不调整现有菜单层级与路由结构
- 不修改后端接口协议与业务逻辑
- 不重做为营销化、重装饰或强看板风格
- 不在第一轮同时覆盖全站所有页面

## 4. 总体推进策略

采用“全站骨架先统一，再按模块逐步落地”的方式推进。

### 4.1 推荐方案

- 第一阶段：统一全局框架与通用页面骨架
- 第二阶段：节点管理、投资模块
- 第三阶段：交易所、ETF、公募基金模块
- 第四阶段：日志管理、统计分析、系统设置、仪表盘

### 4.2 采用该策略的原因

- 可以避免先做的模块和后做的模块产生风格漂移
- 可以先沉淀通用模板，降低后续逐页改造成本
- 符合“按照现有菜单模块一个一个来”的推进要求

## 5. 统一设计原则

### 5.1 风格原则

- 标准后台简洁
- 列表优先
- 弱装饰、低饱和
- 强调主操作，收敛次操作
- 信息分层清晰，避免同屏噪音过多

### 5.2 页面结构原则

所有业务页面优先统一为以下骨架：

- 页面标题区
- 筛选与操作区
- 主内容区
- 分页区
- 弹窗或抽屉区

### 5.3 交互原则

- 查询与重置位置统一
- 新增按钮作为页面主操作固定出现
- 编辑、删除、详情等行内操作顺序统一
- 空状态、加载状态、禁用状态表达统一

## 6. 页面类型规范

### 6.1 标准列表页

适用于：

- 分组管理
- 券商管理
- 交易所列表
- 日志页

结构规范：

- 顶部标题与说明
- 一行或两行筛选栏
- 右侧工具操作区
- 标准表格容器
- 统一分页

### 6.2 卡片型业务页

适用于：

- 节点列表

结构规范：

- 标题区与筛选区
- 卡片网格
- 每张卡片包含主状态、核心资源信息、次级元信息、操作区
- 在保持状态可视化优势的同时控制视觉复杂度

### 6.3 标准详情页

适用于：

- 投资详情
- 券商详情
- 基金详情

结构规范：

- 返回入口与主标题
- 基础信息块
- 关联数据块
- 图表或时序数据块
- 操作与辅助信息区域分离

### 6.4 配置页

适用于：

- 系统设置

结构规范：

- 分组表单
- 清晰保存入口
- 尽量减少无关视觉装饰

### 6.5 首页与分析页

适用于：

- 仪表盘
- 相关统计

结构规范：

- 保留少量概览信息
- 强化主要图表或结果区
- 不与 CRUD 列表页强行统一为同一视觉形式

## 7. 模块推进顺序

## 8. 第一轮落地范围

第一轮只落地 4 个文件，建立整站基线：

- `src/layouts/MainLayout.vue`
- `src/layouts/TagsView.vue`
- `src/pages/node/List.vue`
- `src/pages/investment/List.vue`

### 8.1 MainLayout.vue 设计边界

要做：

- 优化侧边栏配色、激活态、折叠态
- 优化顶部栏、主内容区背景与容器层级
- 统一整站框架气质

不做：

- 不改菜单结构
- 不改路由逻辑
- 不改修改密码、退出登录等业务流程

### 8.2 TagsView.vue 设计边界

要做：

- 优化页签样式、激活态、关闭反馈、间距节奏

不做：

- 不改页签缓存逻辑
- 不改页签关闭行为

### 8.3 node/List.vue 设计边界

要做：

- 建立卡片型业务页样板
- 强化筛选、状态、资源信息、主要操作层级
- 优化卡片布局与响应式适配

不做：

- 不改节点详情、审批、编辑、删除接口逻辑
- 不改后端数据结构

### 8.4 investment/List.vue 设计边界

要做：

- 建立复杂列表页样板
- 统一筛选栏、工具栏、表格层级、抽屉入口节奏

不做：

- 不改投资列表查询逻辑本身
- 不改抽屉业务流程
- 不改详情页路由逻辑

## 9. 预期产出

第一轮完成后，将得到 4 类基础样板：

- 全局框架样板
- 标签导航样板
- 卡片型业务页样板
- 标准复杂列表页样板

后续页面改造以这些样板为基准复用。

## 10. 验证方式

每轮改造完成后统一做以下验证：

- 运行前端构建，确认无编译错误
- 运行 lint 与类型检查
- 检查页面文件诊断信息
- 检查主要页面结构渲染是否正常
- 检查移动端与中等宽度下是否有明显错位
- 确认未破坏原有业务交互

## 11. 下一步

在当前设计确认后，进入实施计划阶段，先为第一轮 4 个文件编写可执行计划，再开始逐个落地与验证。
