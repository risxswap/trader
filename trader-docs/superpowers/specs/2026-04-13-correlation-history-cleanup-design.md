# Correlation History Cleanup Design

## 背景

`correlation` 当前已经切换为 append-only 写入，更新和重算都只插入新版本，不再在写路径执行逐条 `DELETE`。这样可以规避 ClickHouse 高频 mutation 导致的 `TOO_MANY_MUTATIONS`。

但 append-only 会自然积累历史版本。如果完全不清理，表体积会持续增长，后续查询和存储成本都会上升。业务目标是:

- 前台和接口始终只读取最新的相关性
- 历史版本不需要长期保留
- 删除旧版本不能重新回到“每次写入一条 mutation”的模式

## 目标

采用方案 A:

- 写路径继续 append-only
- 读路径继续只读最新版本
- 在全量相关性重算任务结束后，批量清理历史版本
- 清理以“批”为单位发起 mutation，显著减少 mutation 数量

## 非目标

- 不追求同一业务键在任意时刻物理上只保留一条记录
- 不在单次 `update()` 或 `calculateAndSave()` 后立刻清理旧版本
- 不改用 MySQL 保存相关性
- 不在本次设计中引入 tombstone/逻辑删除字段

## 现状问题

### 写路径

当前 `CorrelationService.update()` 和 `calculateAndSave()` 已经改为只追加写入新版本。

### 读路径

当前列表查询和按业务键查询已经只读取 `(asset1, asset2, period)` 的最新版本。

### 缺口

- 旧版本还会留在 ClickHouse 中
- 显式删除接口仍依赖 mutation
- 全量重算任务当前只是把任务提交给线程池，没有“本轮任务全部结束”的收口点

其中最后一点是方案 A 的关键前提: 必须在“本轮重算都完成之后”再触发一次历史清理。

## 设计概览

### 1. 写路径保持 append-only

`update()` 和 `calculateAndSave()` 保持现状:

- 读取当前最新版本作为参考
- 生成新 `id`
- 插入新版本
- 不同步删除旧版本

### 2. 读路径保持 latest-only

查询继续保证:

- `getByUniqueKey(asset1, asset2, period)` 返回最新版本
- 列表分页每组业务键只返回最新版本

这样即使旧版本暂时存在，用户也只会看到最新相关性。

### 3. 新增历史清理能力

新增一个专门的清理流程，只做一件事:

- 对每组 `(asset1, asset2, period)` 保留最新一条
- 删除其余旧版本

清理步骤:

1. 查询存在多版本的业务键
2. 为每组业务键找出需要保留的最新 `id`
3. 收集需要删除的旧 `id`
4. 按批次执行批量删除

### 4. 触发时机

在 `CorrelationTask.doCalculateAllCorrelations()` 中:

1. 提交所有相关性计算任务
2. 等待本轮所有任务执行完成
3. 再触发一次历史清理

这样可以把很多次小删除，合并为一轮有限批次的清理动作。

## 组件设计

### CorrelationMapper

新增或调整以下能力:

- 查询有重复版本的业务键
- 查询每组业务键最新版本 `id`
- 查询旧版本 `id`
- 按一批 `id` 执行批量删除

删除 SQL 不再是一条旧记录一条 mutation，而是批量 `id` 一次 mutation。

### CorrelationDao

封装清理相关的 DAO 方法:

- 查询重复业务键
- 查询某组业务键需要删除的旧 `id`
- 批量删除旧 `id`

DAO 层只负责数据访问，不写清理流程控制逻辑。

### CorrelationService

新增清理入口，例如:

- `cleanupHistoricalCorrelations()`

职责:

- 分页/分批遍历重复业务键
- 对每组业务键定位旧版本
- 聚合待删除 `id`
- 到达批量阈值后调用 DAO 批量删除

建议参数:

- 业务键批大小
- 删除 `id` 批大小

这样可以根据线上数据量逐步调优。

### CorrelationTask

现有任务是把计算工作提交给 `correlationExecutor`，但没有等待完成。

需要改成可等待的模式，例如:

- 使用 `CompletableFuture`
- 或维护 `CountDownLatch`
- 或收集 `Future<?>` 后统一 `get()`

要求:

- 只有本轮所有相关性计算任务都结束后，才进入历史清理
- 如果任务执行中有异常，任务日志中需要可见
- 清理过程也要写入任务日志，便于观察本轮清理了多少旧版本

## 数据流

### 重算流程

1. 全量任务提交基金对计算任务
2. 每个任务只追加写入最新版本
3. 主任务等待所有计算任务结束
4. 主任务调用历史清理服务
5. 清理服务批量删除旧版本

### 手工更新流程

1. `PUT /correlation` 只追加写新版本
2. 查询仍只看到最新版本
3. 历史版本留给后续全量重算后的批处理清理

## 错误处理

- 如果单个相关性计算失败，不应阻塞其他组合的计算
- 如果等待任务完成时发现异常，需要记录日志并继续决定是否执行清理
- 如果清理批次失败，需要记录失败批次大小和上下文，不应吞异常
- 若清理失败，最多只会留下旧版本，不应影响“读最新”的正确性

## 测试策略

### 服务测试

- `update()` 追加写入新版本且不删除旧版本
- `calculateAndSave()` 追加写入新版本且不删除旧版本
- 清理服务只删除旧 `id`，保留最新 `id`

### Mapper/DAO 测试

- 校验 latest-only 查询 SQL 存在
- 校验批量删除 SQL 存在
- 校验重复业务键聚合查询存在

### 任务测试

- 验证全量任务会等待所有计算任务结束
- 验证等待完成后会触发一次清理入口

## 风险与缓解

### 风险 1: 清理批次过大

如果一批删除的 `id` 太多，单次 mutation 会变重。

缓解:

- 采用小批次删除，例如 100 或 200 个 `id`
- 保持清理轮次少于逐条删除产生的 mutation 数量

### 风险 2: 任务尚未结束就开始清理

这会导致正在写入的新版本和清理过程相互交错。

缓解:

- 必须先补“等待本轮任务完成”的收口点

### 风险 3: 显式删除接口仍走 mutation

当前 `delete(Long id)` 仍是直接删除。

缓解:

- 本次先解决批量重算主路径
- 后续如果要彻底去 mutation，再单独设计显式删除的替代方案

## 结论

方案 A 的核心不是完全避免删除，而是:

- 平时写入不删
- 查询永远读最新
- 删除只在重算任务结束后集中、分批执行

这样既满足“只保留最新相关性”的业务目标，也避免重新回到每次更新一条 mutation 的 ClickHouse 反模式。
