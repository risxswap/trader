# 任务创建弹窗下拉可选性调整设计

## 背景

当前任务管理页的“创建任务实例”弹窗中：

- 下拉框直接使用 `/task/definition/list` 返回的全部任务定义
- 前端不区分哪些任务定义已经创建过实例
- 前端也不区分策略类型与非策略类型

因此当前存在两个体验问题：

1. 非策略任务如果已经创建过实例，用户仍能在下拉里再次选择，最终只能等后端报错
2. 策略类型任务也会出现在创建下拉中，但当前需求希望在该下拉中隐藏策略类型

用户要求：

- 创建任务时，下拉选择任务实例
- 已经存在的任务实例置灰，不让选择
- 隐藏策略类型的任务
- 仅作用于“创建任务实例”弹窗，不影响任务列表和已有实例展示

## 目标

本次改造目标如下：

1. 在创建弹窗中隐藏 `taskType=STRATEGY` 的任务定义
2. 对非策略任务中已经存在实例的定义项进行置灰展示
3. 保持后端现有“非策略任务不能重复创建”的兜底规则不变
4. 不扩大为后端协议重构，仅在前端页面完成展示和交互优化

## 非目标

- 不修改 `/task/definition/list` 的接口结构
- 不修改 `TaskDefinitionDto`
- 不修改 `SystemTaskController` 与 `SystemTaskService.definitions()`
- 不隐藏任务列表中已经存在的策略实例
- 不改变任务详情、任务列表筛选、任务执行等逻辑

## 当前现状

### 现状一：创建下拉只使用 definitions 数据

`admin-web/src/pages/task/Manage.vue` 中创建弹窗的下拉：

- 直接遍历 `definitions`
- 选项 value 为 `${taskType}::${taskCode}`
- 没有 disabled 状态
- 没有“已创建”文案

### 现状二：前端已持有当前页面任务实例数据

同一页面已经通过 `loadData()` 拉取当前任务列表：

- 数据存放在 `tableData`
- `tableData` 中已经包含现存任务实例的 `taskType + taskCode`

这意味着前端无需新增后端接口，即可本地判断：

- 某个任务定义是否已经存在实例

### 现状三：后端已有兜底

后端 `SystemTaskService.createInstance()` 当前已经具备：

- 非 `STRATEGY` 类型：若同 `taskType + taskCode` 已存在，则拒绝创建
- `STRATEGY` 类型：允许重复创建并生成实例码

因此本次前端改造主要是：

- 提前把不可选项表现出来
- 减少用户点完“创建任务”才得到后端错误的情况

## 总体设计

### 设计原则

- 规则主要放在创建弹窗前端展示层实现
- 后端规则不删除，保留安全兜底
- UI 改动尽量限制在 `Manage.vue`
- 不扩大为任务定义接口协议变更

### 采用方案

采用“前端过滤 + 后端兜底”的方案：

- 前端负责隐藏策略类型和置灰已创建非策略定义
- 后端继续拦截非策略重复创建请求

这样既满足用户体验，也避免不必要的后端 DTO 和接口调整。

## 创建下拉的数据计算

### 输入数据

创建下拉使用两份前端已有数据：

1. 任务定义列表：`definitions`
2. 当前页面已加载的任务实例列表：`tableData`

### 计算规则

在 `Manage.vue` 中增加一个专门给创建弹窗使用的计算列表，例如：

- `availableDefinitions`

计算步骤：

1. 从 `definitions` 中过滤掉 `taskType === 'STRATEGY'`
2. 对剩余定义逐个判断是否已存在同 `taskType + taskCode` 的实例
3. 为每个选项补充本地展示态字段：
   - `disabled`
   - `disabledReason`

### 已存在实例的判断规则

仅对非策略任务生效：

- 若 `tableData` 中存在同 `taskType + taskCode` 的项，则视为“该任务定义已创建实例”

注意：

- 本次仅基于当前页面已加载的任务实例列表判断
- 由于创建弹窗只服务当前页面用户操作，这个判断足以满足交互需求

## 展示与交互设计

### 隐藏规则

策略类型任务定义：

- 不出现在创建下拉中

### 置灰规则

非策略且已存在实例的定义项：

- 保留在下拉中
- 设置为 `disabled`
- 不允许选中

### 文案规则

为了让用户理解为什么不可选，在 label 上直接追加提示：

- `任务名（taskType/taskCode，已创建）`

例如：

- `同步基金（COLLECTOR/fundSync，已创建）`

这样可以避免额外 tooltip 或二次交互成本。

### 已选项失效处理

如果弹窗数据刷新后，当前已选中的定义变成 disabled：

- 自动清空 `createForm.taskRef`

避免出现表单里保留一个逻辑上已不可选项。

### 创建按钮提交兜底

创建按钮点击时，前端继续做轻量兜底：

- 若未选择定义：提示 `请先选择任务实例`
- 若选中的定义已被本地标记为 disabled：提示 `该任务实例已创建，不能重复创建`

这层拦截仅用于用户体验，不替代后端安全校验。

## 数据刷新时机

### 打开弹窗

打开“创建任务实例”弹窗时：

- 拉取最新 `definitions()`
- 直接复用当前页面已有的 `tableData`

### 创建成功后

创建成功后：

1. 关闭弹窗
2. 重新执行 `loadData()`
3. 重新加载任务定义

这样下次打开弹窗时：

- 刚创建成功的非策略任务就会自动变成置灰状态

## 实施边界

本次改造主要修改：

- `trader-admin/admin-web/src/pages/task/Manage.vue`

如有需要，可以在前端页面内部补充本地类型扩展，例如：

- `CreateTaskDefinitionOption`

用于携带：

- `disabled`
- `disabledReason`

本次不修改：

- `admin-web/src/services/systemTask.ts` 的接口协议定义
- `admin-server` 任何 DTO、Controller、Service
- 任务列表与详情页展示逻辑

## 验证策略

重点验证四组场景：

### 1. 策略类型定义

- 在创建下拉里不出现

### 2. 非策略且未创建

- 正常显示
- 可以选中
- 可以正常创建

### 3. 非策略且已创建

- 继续显示在下拉中
- 带 `已创建` 提示
- 呈现为 disabled
- 无法选中

### 4. 创建成功后的刷新

- 新创建的非策略任务在重新加载后变成 disabled
- 下次打开弹窗不能再次选择

## 测试策略

本次以页面级行为验证为主：

- 优先补 `Manage.vue` 相关的前端单测或轻量行为测试
- 如果当前项目缺少前端测试基础，则至少做构建验证和手工场景验证

验证重点：

- `availableDefinitions` 的过滤与 disabled 计算
- 创建提交前 disabled 项兜底提示

## 验收标准

- 创建弹窗下拉中不再显示策略类型任务定义
- 非策略已创建任务定义显示为置灰且不可选
- 下拉项能明确展示“已创建”状态
- 前端无法通过正常交互再次选择已创建的非策略任务
- 后端现有重复创建兜底逻辑保持有效
