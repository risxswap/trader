# Trader Admin 需求分析

## 项目概述
- 目标：构建用于管理投资项目、策略执行与交易日志的后台系统，并支持将交易报告推送到外部消息渠道（企业微信）。
- 技术栈：后端 Spring Boot + MyBatis-Plus + MySQL；前端 Vue 3 + Vite + Element Plus；消息推送集成企业微信。
- 分层架构：Controller（REST 接口与参数接收）→ Service（业务逻辑）→ Dao（数据库操作）；异常统一处理与统一响应结构（code/msg/data）。

## 目标与范围
- 投资项目管理：维护项目基本信息、预算、标的、策略配置、执行器与调度。
- 策略执行与调度：按 cron 定时触发策略，产生日志与报表。
- 交易日志记录与查询：记录资金与资产变动，支持条件查询与导出。
- 消息推送：以 Markdown 渲染的交易报告推送至企业微信机器人。
- 基础设置：目标标的类型/分组、券商账户、执行器配置等。

## 架构与约束
- 约束：
  - 严格分层：Controller 不写业务；Service 不写 SQL；Dao 负责 SQL。
  - 参数抽象：查询参数超过 3 个使用 *Query*；更新/编辑参数超过 3 个使用 *Param*；响应数据字段超过 2 个使用 *Dto*。
  - 数据库命名规范：下划线命名；主键 id；外键 ${表名}_id；索引 ${表名}_${字段名}_idx。
  - 异常：ErrorCode 枚举 + Warning 抛出 + ExceptionAdvice 统一返回。
- 非功能：
  - 安全：敏感配置通过环境变量/配置文件管理，不入库不日志。
  - 性能：必要索引，调度幂等，避免重复执行与重复入库。
  - 监控：Actuator 健康检查；推送与执行结果审计日志。

## 核心模块
- 投资项目管理（Investment）
  - 维护 name、group_name、target_type、invest_type、broker_id、targets、budget、strategy、strategy_config、cron、executor_id、status 等。
  - 支持 CRUD、分页查询、按分组/状态过滤；策略配置读写；调度启停。
- 策略执行与调度
  - 基于 cron 周期触发指定 strategy，由指定 executor 执行；生成交易日志。
  - 执行前校验配置完备性，执行后记录日志与推送。
- 交易日志（InvestmentLog）
  - 记录 investment_id、type（TRADE/ADJUST/REBALANCE 等）、record_date、cash、asset 等。
  - 支持按项目/类型/时间范围查询与导出。
- 消息推送（企业微信）
  - 生成交易报告 Markdown，推送到指定的企业微信群机器人。
  - 推送日志落库，用于审计与重试。

## 模块细化
- 投资项目模块
  - 接口
    - POST /api/investments（创建，Body: InvestmentParam）
    - PUT /api/investments/{id}（更新，Body: InvestmentParam）
    - DELETE /api/investments/{id}（删除）
    - GET /api/investments（分页查询，Query: InvestmentQuery，支持 name/groupName/status）
    - PUT /api/investments/{id}/status（启停，Body: Param 包含 status 与校验结果）
  - 校验
    - name 必填，最大 64
    - cron 为合法表达式；当 status=RUNNING 时必须存在且合法
    - targets 非空时应与 target_type 一致（如股票代码/基金代码列表），类型为 List<String>
    - budget ≥ 0，numeric 精度符合财务要求
    - strategy 为受控字典值；strategy_config 为 JSON 字符串或 text
    - executor_id 当 RUNNING 时必填，用于定位执行器
  - 业务规则
    - 状态流转：STOPPED → RUNNING 需校验 cron/executor/strategy_config 完备；RUNNING → STOPPED 立即停止调度
    - 更新敏感字段（strategy/cron/executor_id）在 RUNNING 时需要额外校验或先 STOPPED
    - 对 targets 的写入与展示遵循 jsonb 存储与 DTO 映射
- 策略执行与调度模块
  - 流程
    - 调度触发：根据 cron 获取需执行的 investment，加载 strategy 与 strategy_config
    - 执行：由 executor 执行交易逻辑，生成执行快照（现金/资产变动）
    - 记录：落库 InvestmentLog，包含 investment_id/type/record_date/cash/asset
    - 推送：调用消息模块生成报告并发送至企业微信
  - 幂等
    - 同一 investment 在同一时间窗执行使用幂等键（如日期+投资ID）防重复入库与重复推送
  - 风险控制（可扩展）
    - 对波动率、仓位上限、单次交易额度做基础校验，超限则记录警告并调整执行
- 交易日志模块
  - 查询
    - 条件：investmentId、type（TRADE/ADJUST/REBALANCE）、dateRange、分页与排序（record_date DESC）
    - 导出：提供 CSV/JSON 导出接口（服务端流式输出）
  - 指标扩展
    - 可增加 pnl、drawdown、risk_flag、notes 等字段，用于数据分析与审计
- 消息推送模块（企业微信）
  - 消息格式
    - msgtype: markdown
    - markdown: 服务端将交易数据渲染为 Markdown（标题/加粗/代码/列表/引用/表格）
  - 异常处理
    - 发送失败时记录错误码与响应原文，标记 FAIL
  - 推送日志
    - 记录渠道、标题、类型、内容摘要、状态、recipient（如 Webhook URL 摘要）、错误信息，支持后续审计与重试
- 前端模块
  - 投资项目列表页
    - 列：名称、分组、策略、状态、预算、执行器、更新时间
    - 功能：筛选/分页、启停按钮、进入详情
  - 项目详情页
    - 区块：基本信息、策略配置编辑（strategy/strategy_config）、targets 编辑、调度设置（cron/executor_id）、状态切换
    - 校验提示：必填项与格式校验，保存前整体校验
  - 交易日志页
    - 列：日期、类型、现金、资产、备注；支持按项目/日期范围筛选；图表展示资产与现金趋势
    - 导出：CSV/JSON
  - 推送记录页
    - 列：时间、渠道、标题、状态、房间、event_id/错误码；查看原文与重试入口（可选）
  - 节点管理页
    - 列：节点ID、节点名称、类型、状态、IP、当前CPU、当前内存、心跳时间
    - 数据源：列表及状态来自Redis，历史资源使用率来自ClickHouse
  - 交互规范
    - 表单遵循后端 validation；响应式布局；错误统一 toast 与表单内提示
- 权限与审计（可选）
  - 角色：Admin（全权限）、Operator（执行与修改）、Viewer（只读）
  - 审计：关键操作与推送落库，支持按时间与操作人检索

## 接口规范与示例
- 统一响应结构：{"code":0,"msg":"OK","data":{...}}
- 示例：创建投资项目
  - Request（InvestmentParam）
    - id（可空，更新时必填）
    - name、groupName、targetType、investType、brokerId
    - targets（List<String>）、budget（numeric）
    - strategy、strategyConfig（JSON 字符串或 text）
    - cron、executorId、status
  - Response
    - data: InvestmentDto（id、基础字段与状态）
- 示例：查询交易日志
  - Query：investmentId、type、startDate、endDate、page、size、sort
  - Response：分页对象 + InvestmentLogDto 列表（recordDate、cash、asset、type）
- 示例：企业微信推送
  - Request：type、title、markdownContent
  - Response：event_id 或错误码与错误信息

## 策略字典与 strategy_config JSON Schema
- 策略字典（示例）
  - MA_CROSS（双均线趋势跟踪）
  - VOL_TARGET（波动率目标）
  - MEAN_REVERSION（均值回归）
- MA_CROSS
  - 字段：shortWindow、longWindow、maxPosition、riskLimit
  - JSON Schema
    ```json
    {
      "$schema": "https://json-schema.org/draft/2020-12/schema",
      "type": "object",
      "required": ["shortWindow", "longWindow", "maxPosition"],
      "properties": {
        "shortWindow": { "type": "integer", "minimum": 1, "maximum": 200 },
        "longWindow": { "type": "integer", "minimum": 2, "maximum": 400 },
        "maxPosition": { "type": "number", "minimum": 0, "maximum": 1 },
        "riskLimit": { "type": "number", "minimum": 0 }
      }
    }
    ```
- VOL_TARGET
  - 字段：targetVol、lookbackDays、maxLeverage、rebalanceFreq
  - JSON Schema
    ```json
    {
      "$schema": "https://json-schema.org/draft/2020-12/schema",
      "type": "object",
      "required": ["targetVol", "lookbackDays"],
      "properties": {
        "targetVol": { "type": "number", "minimum": 0.01, "maximum": 0.5 },
        "lookbackDays": { "type": "integer", "minimum": 5, "maximum": 365 },
        "maxLeverage": { "type": "number", "minimum": 1, "maximum": 5 },
        "rebalanceFreq": { "type": "string", "enum": ["DAILY", "WEEKLY", "MONTHLY"] }
      }
    }
    ```
- MEAN_REVERSION
  - 字段：lookbackDays、entryThreshold、exitThreshold、stopLoss
  - JSON Schema
    ```json
    {
      "$schema": "https://json-schema.org/draft/2020-12/schema",
      "type": "object",
      "required": ["lookbackDays", "entryThreshold", "exitThreshold"],
      "properties": {
        "lookbackDays": { "type": "integer", "minimum": 5, "maximum": 365 },
        "entryThreshold": { "type": "number", "minimum": 0.5, "maximum": 5 },
        "exitThreshold": { "type": "number", "minimum": 0.1, "maximum": 5 },
        "stopLoss": { "type": "number", "minimum": 0, "maximum": 0.3 }
      }
    }
    ```
- 校验与展示
  - 后端保存时校验 strategy 与 strategy_config 字段符合 schema
  - 前端提供 JSON 编辑器与 schema 校验提示，显示字段的范围、枚举与依赖关系

## 对象定义（建议）
- InvestmentParam
  - id: Integer；name: String；groupName: String；targetType: String；investType: String
  - brokerId: Integer；targets: List<String>；budget: BigDecimal
  - strategy: String；strategyConfig: String；cron: String；executorId: String；status: String
- InvestmentQuery
  - name: String；groupName: String；status: String；page: Integer；size: Integer；sort: String
- InvestmentDto
  - id、name、groupName、strategy、status、budget、executorId、updatedAt
- InvestmentLogDto
  - id、investmentId、type、recordDate、cash、asset、notes（可选）

## 错误码约定（示例）
- 0：SUCCESS
- 40001：PARAM_INVALID（参数非法）
- 40004：NOT_FOUND（资源不存在）
- 40300：FORBIDDEN（无权限或房间限制）
- 50000：INTERNAL_ERROR（系统异常）
- 51001：WECOM_FORBIDDEN（企业微信权限错误）
- 51002：WECOM_CONFIG_ERROR（配置错误）
- 51003：WECOM_UNKNOWN（企业微信未知错误）

## 校验规则细化
- name 非空且 ≤64；groupName ≤64
- budget 为正数，scale 与 precision 按财务场景约定（如 precision=18, scale=4）
- cron 需符合 Quartz 表达式；建议时区统一为系统时区或配置项
- strategy 为受控值（如：MA_CROSS、MEAN_REVERSION、VOL_TARGET 等）；strategyConfig 为相应 JSON schema
- targets 与 targetType 映射一致，如 targetType=STOCK 时 targets 为股票代码列表

## 调度与执行细节
- 时区：统一使用服务端时区或配置定义；记录时间一律使用 timestamptz
- 并发控制：同一 investment 在同一触发时间只执行一次；使用分布式锁或幂等键
- 失败重试：执行失败记录原因，不自动重试；由人工或策略控制重试

## 幂等与重试
- 幂等键：investmentId + yyyy-MM-dd（或更细粒度）作为执行快照唯一键
- 推送重试：仅在 join 成功后重试一次；避免形成推送风暴

## 审计与索引
- 建议索引
  - investment(name、group_name、status、updated_at)
  - investment_log(investment_id、record_date、type)
  - msg_push_log(recipient、created_at、status)
- 审计字段：created_at、updated_at 必填，默认 CURRENT_TIMESTAMP

## 企业微信推送格式示例

## 字段占位示例与错误提示文案
- InvestmentForm 占位
  - name: 请输入项目名称（≤64）
  - groupName: 请输入分组（≤64）
  - targetType: 请选择标的类型
  - investType: 请选择投资类型
  - brokerId: 请输入券商账户ID
  - targets: 请选择/输入标的代码列表
  - budget: 请输入预算金额（≥0）
  - strategy: 请选择策略（如 MA_CROSS）
  - strategyConfig: 请输入 JSON 配置（支持校验）
  - cron: 请输入调度表达式（RUNNING 必填）
  - executorId: 请输入执行器ID（RUNNING 必填）
- 错误提示文案
  - name: 名称不能为空，且不超过 64 字符
  - brokerId: 券商账户不能为空
  - budget: 预算需为非负数字
  - cron: 调度表达式不合法
  - executorId: 执行器ID不能为空（RUNNING）
  - strategy: 策略不能为空或不在字典中
  - strategyConfig: 策略配置不合法或缺少必填字段
  - targets: 标的列表格式不正确或与类型不一致

## 数据模型（数据库与实体）
- Investment（public.investment）
  - 关键字段：id、name、group_name、target_type、invest_type、broker_id、targets(jsonb)、budget(numeric)、strategy、strategy_config(text)、cron、executor_id、status、created_at、updated_at
  - 说明：targets 以 jsonb 列表存储；遵循 MySQL 规范与范式。
- InvestmentLog（public.investment_log）
  - 关键字段：id、investment_id、type、record_date、cash、asset（可扩展盈亏与风险指标）。
- 消息推送日志（MsgPushLog）
  - 记录 type、title、content、status、channel、recipient、created_at、updated_at 与响应信息。

## 接口需求（后端）
- 投资项目
  - 创建/更新/删除/分页查询；按分组与状态过滤。
  - 策略配置：strategy 与 strategy_config 的读写。
  - 调度管理：设置/更新 cron；启停（status：RUNNING/STOPPED）。
  - 目标管理：targets（jsonb）写入与校验。
- 交易日志
  - 写入：策略执行后落库，关联 investment_id。
  - 查询：按日期范围/类型/项目筛选；导出 CSV/JSON（可后续扩展）。
- 消息推送
  - 推送：将交易报告转换为 HTML（formatted_body）并发送至企业微信。
  - 加入与重试：遇到 M_FORBIDDEN 且提示未加入时尝试 join；公共房间自动 join，私有房间需邀请。
  - 状态落库：记录 event_id、错误码与响应原文。

## 前端需求（Vue 3 + Element Plus）
- 页面
  - 投资项目列表：筛选、分页、状态与快捷启停。
  - 项目详情：基本信息、策略配置编辑、targets 可视化编辑、调度设置。
  - 交易日志：表格与图表视图，支持筛选与导出。
  - 推送记录：查看发送状态、响应详情与重试入口（可选）。
- 交互与校验
  - 表单校验：配合后端 validation，遵循 Dto/Param/Query 规范。
  - 响应式布局：PC/Mobile 适配。
- 可视化
  - 使用 Element Plus Table；若需图表可评估 ECharts 展示资产/现金趋势与盈亏。

## 业务规则
- 调度与状态
  - RUNNING：按 cron 触发；STOPPED：不触发。
  - 执行前校验：cron、executor_id、strategy_config 必备。
- 日志规则
  - 同一 investment_id 可多条记录；record_date 使用执行时间戳。
  - 回补记录需标注 type（如 ADJUST/REBALANCE），避免与日常 TRADE 混淆。
- 推送规则
  - 格式：msgtype markdown + markdown
  - 私有房间需先邀请机器人；公共房间支持自动 join。
  - 失败标记并落库，支持后续人工重试。

## 配置需求
- 企业微信：webhook-url 必填。
- 数据库：遵循 db.sql 建表；按查询场景建立必要索引。
- 应用参数：策略字典与默认配置集中管理，避免硬编码分散。
